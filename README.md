# Candidate Dashboard

HR-дашборд для просмотра кандидатов, анализа соответствия вакансии и управления статусами резюме.

## Стек

- **React 18** + **TypeScript** (строгая типизация, без `any`)
- **Vite** — сборка
- **Zustand** — state management (легковесная альтернатива Redux Toolkit)
- **React Router v6** — маршрутизация
- **Tailwind CSS** — стилизация
- **react-window** — виртуализация списка
- **react-hot-toast** — уведомления
- **Jest + React Testing Library** — тестирование

## Запуск

```bash
npm install
npm run dev        # локальный сервер http://localhost:5173
npm run test       # запуск тестов
npm run build      # продакшен сборка
npm run lint       # ESLint проверка
```

## Функциональность

### Задание 1 — Список кандидатов

- Карточки с ФИО, городом, опытом, вердиктом, стеком, статусом
- Фильтрация по вердикту (Все / ПОДХОДИТ / ЧАСТИЧНО / НЕ ПОДХОДИТ)
- Поиск по ФИО с debounce 300ms
- Сортировка по имени, опыту, дате добавления (возрастание/убывание)
- Пагинация по 10 кандидатов
- Состояние фильтров сохраняется в URL query params
- Адаптивная вёрстка (1/2/3/4 колонки)
- Обработка пустого результата

### Задание 2 — Детальная карточка кандидата

- Маршрут `/candidate/:id`
- Полная информация: контакты, опыт работы, образование, стек
- Критерии оценки с цветовой индикацией (ok/partial/no)
- Вопросы для собеседования
- Summary
- Кнопка «Назад» сохраняет фильтры через query params
- 404-обработка (кандидат не найден)

### Задание 3 — Управление статусами

- Выпадающий список: Новый → На рассмотрении → Приглашён → Отклонён
- Mock API запрос `PATCH /api/candidates/:id/status`
- **Оптимистичное обновление**: статус меняется мгновенно
- **Rollback**: при ошибке API статус откатывается к предыдущему
- Loading spinner во время запроса
- Toast-уведомления об успехе/ошибке (react-hot-toast)
- Статус отображается в списке (StatusBadge)

### Задание 4 — Тестирование

- Unit-тесты: фильтрация, сортировка, debounce, VerdictBadge, StatusBadge, FilterPanel, Pagination
- Integration-тест: навигация из списка в детальную страницу
- Edge cases: пустой результат поиска, граничные значения пагинации

### Задание 5 — Оптимизация производительности

- Данные берутся из `candidates-large.json` (120 кандидатов)
- **Виртуализация** через `react-window` (FixedSizeGrid) при списке > 30 элементов
- `useMemo` для фильтрации и сортировки
- `React.memo` для тяжёлых компонентов (CandidateCard, FilterPanel, VerdictBadge, StatusBadge, Pagination)
- `ResizeObserver` для адаптивной ширины виртуального списка
- Cleanup в useEffect (ResizeObserver disconnect, debounce clearTimeout)

## Архитектурные решения

**Zustand вместо Redux Toolkit** — для дашборда такого масштаба Zustand даёт тот же результат с в 3 раза меньшим boilerplate. Если проект вырастет до нескольких слайсов с complex selectors — переход на RTK тривиален.

**URL-параметры для состояния фильтров** — позволяет делиться ссылкой на конкретный фильтр и не теряет состояние при перезагрузке.

**Оптимистичное обновление** — реализовано на уровне Zustand store: `updateStatus` → API call → при ошибке `rollbackStatus`. Это чище, чем хранить prevState в компоненте.

**Виртуализация с порогом** — для малых списков (≤30) используется обычный grid с пагинацией (лучше для SEO и a11y). Виртуализация включается только при большом наборе данных.
